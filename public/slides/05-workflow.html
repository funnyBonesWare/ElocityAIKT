Our AI-Powered Software Workflow

Every feature, bug fix, and improvement follows this end-to-end pipeline — from a Jira ticket to production-ready code with automated evidence. Here's exactly how it works, step by step.

The Pipeline — End to End

Each Step — In Detail

Step 1: Jira Ticket
Who: CTO (Dinesh) or Product Team creates the task in Jira.
What: A clear requirement — feature, bug fix, or improvement with acceptance criteria.
Example: "Build charger analytics dashboard with date filters, station-wise breakdown, and export."
AI role here: None yet — this is a human decision about what to build.
Step 2: Research & Context
Who: Engineer picks up the ticket and gathers context.
What: Read Confluence docs, existing codebase, API docs, design specs. AI helps summarize and find patterns.
Tools: Confluence, AI chat (Claude/GPT) for Q&A about the codebase and architecture decisions.
Output: A mental model + approach — the engineer knows what to build and how.
Step 3: AI-Assisted Coding ⭐
Who: Engineer + Cursor AI Agent working together.
What: The engineer describes the task in Cursor. The AI agent generates code — React components, API integrations, database queries, business logic — across multiple files.
How it works: Engineer guides the direction. AI writes 70-80% of the code. Engineer reviews, adjusts, and adds domain knowledge the AI can't infer.
Time saved: What used to take 1-2 weeks of coding now takes 2-3 days. The AI handles boilerplate, patterns, and repetitive logic.
Tool: Cursor (AI-native IDE with built-in agent mode).
Step 4: AI Code Review
Who: AI reviewer + human engineer for final approval.
What: Code pushed to GitLab triggers an AI review that checks for security issues, performance problems, code style, and best practices.
What it catches: Hardcoded secrets, missing error handling, N+1 queries, accessibility issues, type mismatches.
Human role: Engineer reads the AI review summary, verifies flagged items, and approves or requests changes.
Tool: GitLab CI/CD + AI review integration.
Step 5: Evidence Loop (Playwright MCP) ⭐
Who: Automated — AI triggers tests via Playwright MCP.
What: End-to-end tests run against the actual app. Playwright opens the browser, navigates pages, clicks buttons, fills forms, and verifies everything works.
Evidence: Screenshots captured at each step. If a test fails, the AI agent sees the failure, fixes the code, and re-runs — all automatically.
Mobile: Maestro runs similar E2E tests on mobile apps.
Output: A complete evidence package — screenshots, test results, pass/fail status — attached to the merge request.
Step 6: Human Verify & Ship
Who: Senior engineer or tech lead — the human gatekeeper.
What: Reviews the AI code review summary, checks the Playwright evidence screenshots, validates business logic makes sense.
Decision: Approve → merge → auto-deploy to staging → final QA → production. Or request changes if something looks off.
Key principle: AI does 90% of the heavy lifting. Humans make the final call. No code ships without a human "yes."

Before AI vs. After AI

Before AI
Ticket → 1-2 days research → 1-2 weeks coding → 2-3 days manual review → 1 day manual testing → deploy
Total: 2-3 weeks per feature
Manual testing = no screenshots, no proof, trust-based QA
Code review = 1 senior engineer, 2-4 hr turnaround
After AI (Now)
Ticket → few hrs AI research → 2-3 days AI coding → same-day AI review → auto E2E tests → human verify → deploy
Total: ~1 week per feature
Playwright MCP = automated screenshots + evidence attached to every MR
Code review = AI flags issues instantly, human just confirms
